# info-serve 限界上下文地图（Bounded Context Map）

状态：v3 定稿（2026-07-03，决策点全部已定）

## 1. 这份文档解决什么问题

0.3.x 定版事故的结构性根源之一，是服务边界从未按领域画过：appcenter 与 infoservice 平级拆分，导致门户公共设施（用户、消息、收藏、统计）被复制两份并已分叉，产生了通知黑洞（论坛回复与资源上新通知写入 `info_portal_message`，全项目无任何读取方）和跨界裸查（`InfoPortalMapper` 直查 `app_application`）。

本文档给出：统一语言、限界上下文（BC）地图、部署归属判据、演进路线与边界纪律。它同时是并行开发（多会话/多 agent）的**分工地图**：每个开发线程认领一个 BC，把冲突面压到最小。

## 4. 部署归属判据与裁决

拆独立服务的正当理由仅四个：**①独立扩缩容 ②技术栈异质 ③发布节奏冲突 ④故障隔离价值**。

| BC | ①扩缩 | ②异质 | ③节奏 | ④隔离 | 裁决 |
|---|---|---|---|---|---|
| 门户内核 | ✗ | ✗ | ✗ | ✗ | 共享模块，永不独立成服务 |
| 工具即用 | ✗ | ✗ | ✗ | ✗ | 并入门户服务（BC 模块） |
| 服务论坛 | ✗ | ✗ | ✗ | ✗ | 门户服务 BC 模块 |
| 时事热点(展示) | ✗ | ✗ | ✗ | ✗ | 门户服务 BC 模块；抓取先用服务内调度 |
| 资料共享 | 暂✗ | ✗ | 可能 | 转换部分✓ | 门户服务 BC 模块，**头号晋升候选**；转换子域先经 RabbitMQ 异步化 |
| 智能问答 | ✓ | ✓✓ | ✓ | ✓ | **出生即独立服务** `ruoyi-ai-qa` |

## 5. 目标形态与演进

```
现在                     阶段1（合并+模块化）           阶段2（按需晋升）
gateway/auth/system/     gateway/auth/system/          gateway/auth/system/
resource(文件)            file(文件·由resource改名)       file(文件)
appcenter   ┐            ruoyi-portal                  ruoyi-portal
infoservice ┘             ├─ portal-kernel              ├─ kernel/appcenter/forum/news
                          ├─ portal-appcenter           └─ resources*
                          ├─ portal-resources          ruoyi-ai-qa      ← 新建即独立
                          └─ portal-forum              preview-worker  ← 转换量大时
```

阶段 1 合并要点：同库零数据迁移（`app_*` 表照旧）；网关将 `/appcenter/**` 路由指向合并服务，前端 API 路径兼容零改动；消息统一至一张表（顺带修复通知黑洞，删除 `info_portal_message` 及复制代码）；compose 减一个服务、发布包减一个镜像。

### 入口层与可配置模块（2026-07-03 新增需求）

门户模块将由后台按权限动态配置（启停/排序/可见范围），且未来会新增第二入口"个人工作台"。两条原则：

1. **入口是编排，不是内容域。** 服务门户、个人工作台、管理后台都是对同一批内容 BC 的不同组合视图。入口数 N 与模块数 M 的复杂度应为 N+M 而非 N×M——前提是入口只消费 BC 契约、公共设施只在内核存在一份。工作台没有自有数据：它聚合内核（我的消息、我的收藏）与各 BC 的 `my-*` 查询（我的资料、我的需求、我的话题）。
2. **可配置模块 = 内核的模块注册表 + 每 BC 一份稳定契约。** 注册表条目：模块编码、名称/描述/图标、入口路由、权限码、状态（启用/敬请期待）、排序。每个内容 BC 对应提供：注册表元数据、入口路由、权限码、`my-*` 查询、统计接口、消息 scene。门户首页从注册表渲染（现为硬编码数组 `home/index.vue:293-297`）；"智能问答/时事热点"正是"已注册未启用"状态的天然用例。

实现落点：建 `portal_module` 专表，权限沿用现有角色-权限码机制；**不建议**复用 `sys_menu`（其模型是管理端路由形状，门户卡片需要描述/图片/状态等字段）。前端门户 shell 与工作台 shell 均从注册表 API 渲染，模块页面沿用现有懒加载路由；微前端（qiankun 等）暂不引入，仅在需要接入第三方独立开发的模块时再评估。

这两条需求不改变部署结论，反而强化它：动态模块要求每个 BC 有稳定契约，多入口要求公共设施唯一——都指向"硬边界模块化"，与微服务个数无关。

## 6. 晋升触发器（满足任一再拆，拆前不拆）

- **Resources → 独立服务**：转换/预览负载影响其他模块响应；或笔记+数据权限迭代需要独立发布节奏；或出现专人/专线持续开发。
- **preview-worker → 独立**：转换队列积压成为常态。
- **news-crawler → 独立**：抓取频率/来源规模超出服务内调度承载。

## 7. 边界纪律（构建期强制，不靠自觉）

1. 每 BC 一个 Maven 子模块；依赖方向仅允许 `BC → kernel`，禁止 BC↔BC。
2. 表归属私有：每 BC 只碰自己前缀的表；跨域数据走对方接口或事件。首页统计改为聚合各 BC 接口，消灭跨界裸 SQL。
3. 内核能力参数化：`message(scene, …)`、`favorite(target_type, target_id)`，禁止每域复制一套。
4. 用 maven-enforcer/ArchUnit 将 1、2 固化进 CI——多线程并行开发时只有构建红灯拦得住越界。
5. 每 BC 独立 controller 前缀（`/portal/<bc>/**`）；晋升 = 挪模块 + 加网关路由。
