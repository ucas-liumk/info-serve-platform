# 0.3.6 正式更新包、UTM 升级验证与失败回退设计

**状态：已确认**
**确认日期：2026-07-16**
**发布分支：`release/0.3.6`**
**基线：`origin/main@3f8b19d4`**
**目标版本：`0.3.6`**

## 1. 背景与发布边界

PR #35 已合入 `origin/main`，但根目录 `VERSION` 仍为 `0.3.5`，仓库也没有历史 `v0.3.5` 正式 tag。当前 UTM 环境运行的是 `/home/kylin/info-serve` 下的 0.3.5 部署，实际代码基线为 `f632f96`，并已使用拆分后的五个 portal 服务。仓库审计确认 `f632f96..3f8b19d4` 的非文档运行时变化全部位于 `plus-ui`，因此本次 0.3.6 正式更新只交付门户前端静态资源。

该包不是对任意标称 `0.3.5` 环境开放的通用更新，而是 **UTM `f632f96` 已验证快照专用正式增量**。安装器必须在任何写操作前验证旧 dist 的完整指纹、五个 portal compose 服务、运行容器和关键 API；无法证明精确基线时拒绝安装。本次不修改数据库，不重建后端镜像，不重新发布 Nacos。更早的、尚未完成 SQL/Nacos/五服务拆分迁移的安装必须先走完整迁移，不能使用本包。

## 2. 目标与非目标

### 2.1 目标

1. 将根目录 `VERSION` 定版为 `0.3.6`。
2. 从干净的 release commit 构建 `plus-ui/dist`。
3. 生成带版本、提交号、文件清单和 SHA-256 sidecar 的正式更新包。
4. 升级前自动备份当前 `plus-ui/dist` 与 `VERSION`。
5. 通过临时目录和同文件系统重命名完成静态资源切换，避免半更新状态。
6. 升级后强制重建 `nginx-web`，并执行 Web 与网关健康检查。
7. 任一升级步骤失败时自动调用回退脚本；回退脚本也可独立人工执行。
8. 在 UTM/Kylin ARM64 上验证正常升级、故障自动回退、人工回退和最终升级。

### 2.2 非目标

- 不发布数据库 SQL。
- 不修改 Nacos 配置。
- 不替换后端 JAR 或 Docker 镜像。
- 不包含 12+9 应用离线包。
- 不在 `releases/` 中提交二进制产物。

## 3. 发布策略

采用两阶段不可变发布，目标源码基线固定为 `origin/main@3f8b19d4`，不得在发布分支静默合入更新后的主线：

1. 从最新 `origin/main` 创建 `release/0.3.6`。
2. 提交版本号、脚本、文档与清单修订。
3. 推送 release 分支并创建 release PR；在 release commit 上创建并推送 `v0.3.6-rc.1`。
4. 在干净的 RC tag checkout 中重建前端和更新包，并在 UTM 完成全部验证。
5. 将验证证据补入 PR，经用户 review 后使用 merge commit 合入 `origin/main`，禁止 squash，确保已测试 release commit 是主线祖先。
6. 验证 `git merge-base --is-ancestor <release-commit> origin/main` 后，在同一 release commit 创建并推送 `v0.3.6`。
7. 已测试归档原样晋级为正式产物，并将归档 SHA-256 发布到对应 GitHub Release；若要求从正式 tag 重装配，必须在隔离目录确定性重建并证明内容清单与已测试包一致。

该流程把“RC tag 构建、正式 tag 同 commit 晋级”视为从正式 tag 装包的等价流程：两个 tag 指向完全相同的 release commit，正式 tag 只在 UTM 验证和主线合入后创建。若主线在此期间漂移，不改变本次 release commit；任何希望纳入的新变化必须重新审计增量并重新走 RC。

## 4. 产物结构

正式归档命名：

```text
info-serve-update-0.3.6-<short-sha>-arm64.tar.gz
info-serve-update-0.3.6-<short-sha>-arm64.tar.gz.sha256
```

归档内根目录：

```text
info-serve-update-0.3.6/
├── RELEASE-METADATA.env
├── MANIFEST.sha256
├── README.md
├── payload/
│   └── plus-ui/
│       └── dist/
└── scripts/
    ├── apply-update.sh
    ├── rollback-update.sh
    └── verify-package.sh
```

`RELEASE-METADATA.env` 只允许简单的 `KEY=value`，包含：

- `PACKAGE_VERSION=0.3.6`
- `REQUIRED_BASE_VERSION=0.3.5`
- `REQUIRED_BASE_COMMIT=f632f960b5d3fd1737b658e45fe43b15dd319e19`
- `REQUIRED_BASELINE_ID=utm-f632f96-split-portal`
- `REQUIRED_BASE_DIST_MANIFEST_SHA256=<旧 dist 排序清单哈希>`
- `SOURCE_TAG=v0.3.6-rc.1`
- `SOURCE_COMMIT=<完整提交号>`
- `SOURCE_SHORT_COMMIT=<短提交号>`
- `SOURCE_BASE_COMMIT=3f8b19d4bb87dcf02b36058a73be7991204350a8`
- `NODE_VERSION=<构建 Node 版本>`
- `NPM_VERSION=<构建 npm 版本>`
- `PACKAGE_LOCK_SHA256=<plus-ui/package-lock.json 哈希>`
- `DIST_MANIFEST_SHA256=<新 dist 排序清单哈希>`
- `BUILD_ARCH=arm64`
- `BUILD_TIME_UTC=<UTC 时间>`

`MANIFEST.sha256` 覆盖归档根目录内除自身外的全部文件。外部 `.sha256` sidecar 覆盖压缩归档本身。README 的第一步必须要求用户在解压、执行任何包内脚本前运行 `sha256sum -c <archive>.sha256`；解压后由独立、只读的 `verify-package.sh` 校验内部清单，不能把升级脚本自验当成执行前信任根。

## 5. 脚本职责

### 5.1 `package-update.sh`

位置：`deploy/scripts/package-update.sh`。

职责：

1. 要求位于干净的 `v0.3.6-rc.1` checkout，`VERSION=0.3.6`，HEAD 与 tag 可解析且父基线为固定的 `3f8b19d4`。
2. 删除旧 `plus-ui/dist`，执行锁文件一致安装和构建（`npm ci`、测试、lint、design audit、production build），禁止复用陈旧 dist。
3. 装配前后再次确认 Git 工作区干净，并核对 `f632f96..SOURCE_COMMIT` 的运行时差异只落在批准的 `plus-ui` allowlist 与 release 脚本/文档内。
4. 接收并校验从目标 UTM 只读采集的旧 dist 排序清单，将其 SHA-256 写入精确基线元数据。
5. 将前端产物、升级脚本、回退脚本、独立校验器和 README 复制到临时装配目录。
6. 记录 tag、完整 commit、固定 base commit、Node/npm 版本、lockfile SHA 与新旧 dist manifest SHA。
7. 生成内部 SHA-256 清单、tar.gz 与外部 SHA-256 sidecar。
8. 默认输出到被忽略的 `releases/` 目录，不把归档加入 Git。

### 5.2 `apply-update.sh`

调用形式：

```bash
sudo ./scripts/apply-update.sh --target /home/kylin/info-serve
```

执行顺序：

1. 校验参数、目标目录、Docker/Compose 和源版本；要求调用者已成功运行独立包校验器。
2. 在任何写操作前验证精确基线：`VERSION=0.3.5`，旧 dist 完整排序清单哈希等于 `REQUIRED_BASE_DIST_MANIFEST_SHA256`，五个 portal compose service 文件存在且对应容器运行，首页与验证码 API 健康。若目标已有匹配 `REQUIRED_BASELINE_ID` 的 `.release-info` 可作为辅助证据，但不能替代 dist 与拓扑检查。
3. 获取排他锁，阻止并发升级或回退；锁冲突必须快速失败。
4. 创建 `.update-backups/0.3.6-<UTC 时间>-<short-sha>/`，用 `cp -a` 备份 `plus-ui/dist`、`VERSION` 和原 `.release-info`。元数据明确记录 `.release-info` 原本是否存在。
5. 生成覆盖备份全部文件的排序 SHA-256 清单；全部完成后原子创建 `BACKUP_COMPLETE`。不完整备份不得参与 `latest` 选择。
6. 将新 dist 复制到目标目录同文件系统的 staging 目录；VERSION 和 `.release-info` 也通过临时文件加 rename 写入。
7. 将旧 dist 重命名为事务临时目录，再把 staging 原子切换为正式 dist。
8. 执行 `(cd "$TARGET/deploy" && docker compose --env-file .env -f docker-compose.yml up -d --no-deps --force-recreate nginx-web)`，记录旧/新容器 ID 与 StartedAt。
9. 检查：
   - `http://127.0.0.1:7010/` 返回 2xx/3xx；
   - `http://127.0.0.1:7010/prod-api/auth/code` 返回 HTTP 200；
   - `nginx-web` 容器为 running。
10. 备份完成后的任一步失败进入一次性错误处理：先 `trap - ERR`，设置 `ROLLBACK_IN_PROGRESS=1`，由当前持锁进程调用不再次加锁的内部 `restore_from_backup`。禁止子进程重新获取同一锁，禁止递归回退。自动回退成功仍以原升级错误码非零退出；回退失败使用独立退出码并保留现场。

升级成功后保留备份，不自动删除。

### 5.3 `rollback-update.sh`

调用形式：

```bash
sudo ./scripts/rollback-update.sh \
  --target /home/kylin/info-serve \
  --backup latest
```

职责：

1. 校验目标目录和指定备份；`latest` 只解析到时间最新、具有 `BACKUP_COMPLETE` 且清单校验通过的备份。
2. 获取与升级脚本相同的排他锁；锁冲突快速失败。
3. 通过共享库调用 `restore_from_backup`：使用同文件系统 staging+rename 原子恢复完整 dist，VERSION 与 `.release-info` 使用临时文件+rename；若备份元数据表明原 `.release-info` 不存在，则删除升级新建的该文件。
4. 执行 `(cd "$TARGET/deploy" && docker compose --env-file .env -f docker-compose.yml up -d --no-deps --force-recreate nginx-web)`，记录旧/新容器 ID 与 StartedAt。
5. 执行写死的正式健康检查，不继承故障注入状态。
6. 记录回退时间、来源备份与结果，不删除失败版本或备份证据。

## 6. 错误处理与安全约束

- 脚本使用 `set -Eeuo pipefail`，错误不得静默吞掉。
- 所有路径先做绝对化和目录边界检查，拒绝 `/`、空路径和非 info-serve 目标。
- 不读取、打印或打包 `.env` 内容。
- 不执行来源不明的包；内外两层 SHA-256 均必须验证。
- 不使用 `rm -rf` 清理用户给定路径；仅清理脚本自行创建且通过前缀校验的 staging 目录。
- 升级与独立回退共用排他锁，但自动回退由持锁的升级进程直接调用无锁恢复函数。
- 自动回退失败时保留现场并输出手工恢复命令，禁止继续覆盖。
- 升级与回退写入 `$TARGET/.update-logs/<operation-id>.log` 和结构化结果元数据；日志禁止包含 `.env`，最终报告记录日志 SHA-256。

## 7. UTM 验证方案

目标环境：`kylin@192.168.64.5`，`aarch64`，Kylin V10 SP1，Docker 26.1.3，Compose 5.2.0。

### 7.1 升级前基线

- 记录 `VERSION`、完整 dist 排序清单及其 SHA-256、Web 容器 ID/启动时间，并生成精确基线指纹。
- `docker compose config --quiet`。
- 冒烟检查首页、验证码接口和资料共享路由。

### 7.2 故障自动回退

仅在显式设置 `INFO_SERVE_ALLOW_FAILURE_INJECTION=1` 时允许 `--test-fail-after-switch`。该开关只在备份完成、dist/VERSION 切换和 nginx 重建后注入一次失败；自动与人工回退始终使用写死的 7010 首页和验证码检查，不读取故障注入参数。验收：

- 升级命令非零退出；
- 自动回退被调用；
- `VERSION` 恢复为 `0.3.5`；
- 完整 dist 排序清单与升级前一致；
- Web 和验证码接口恢复；
- 备份和失败日志保留。

故障注入必须写入审计日志，正式默认不可触发。

### 7.3 人工回退

先正常升级到 0.3.6，再执行 `rollback-update.sh --backup latest`。验收旧版本、旧哈希与服务健康全部恢复。

### 7.4 最终升级

再次执行正常升级并保留 0.3.6：

- `VERSION=0.3.6`；
- 目标 dist 与包内 dist 清单一致；
- 首页、验证码接口、资料共享页与文件预览路由可访问；
- 浏览器控制台无新增错误；
- 所有原有核心容器仍运行；
- 从升级开始 UTC 时间执行 `docker logs --since <UTC>`，确认 nginx、gateway 和五个 portal 服务无新增未解释的 `ERROR`、`Exception`、`FATAL`、`panic`；既有日志不计入本次失败。

提交与定版前必须通过：

```bash
git diff --check
cd plus-ui && npm run design:audit
cd plus-ui && npm run test
cd plus-ui && npm run lint
cd plus-ui && npm run build:prod
bash -n deploy/scripts/package-update.sh
```

`deploy/tests/update-package-test.sh` 使用临时 target 与 fake docker/curl，覆盖参数缺失、路径逃逸、错误版本、内外清单篡改、部分备份、`latest` 完整筛选、锁冲突快速失败、切换中断、一次性自动回退、回退失败不递归、原 `.release-info` 缺失恢复和健康检查失败。仅 `bash -n` 不算行为验证。

## 9. 交付与版本控制

1. 提交信息使用 conventional 中文格式，不附署名尾注。
2. 推送 `release/0.3.6`，创建 release PR，并推送 `v0.3.6-rc.1`。
3. UTM 验证后把证据补入 PR；用户 review 后使用 merge commit 合入主线。
4. 合并前检查 `git log origin/main..HEAD` 与 `git diff origin/main...HEAD`；合并后验证 release commit 是 `origin/main` 祖先，再推送 `v0.3.6`。
5. 在 GitHub Release 发布正式归档 SHA-256 与 tag commit；README 要求同时核对二者。
6. `releases/` 产物永不入库。
7. 最终报告包含包路径、大小、SHA-256、UTM 命令与结果、回退证据、访问地址、审计日志 SHA 和残余风险。

## 10. 验收标准

- 正式 `v0.3.6` 与经过 UTM 验证的 release commit 一致。
- 更新包只在通过完整指纹与五服务拓扑校验的 UTM `f632f96` 基线上升级到 0.3.6；其他 0.3.5 环境明确拒绝。
- 健康检查失败能自动恢复到升级前文件和版本。
- 独立回退脚本能再次恢复旧版本且服务健康。
- 最终 UTM 状态停留在 0.3.6。
- 更新归档与 sidecar 哈希一致，Git 工作区不包含发布二进制。
