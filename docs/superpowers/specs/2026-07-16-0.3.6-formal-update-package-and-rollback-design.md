# 0.3.6 正式更新包、UTM 升级验证与失败回退设计

**状态：已确认**
**确认日期：2026-07-16**
**发布分支：`release/0.3.6`**
**基线：`origin/main@3f8b19d4`**
**目标版本：`0.3.6`**

## 1. 背景与发布边界

PR #35 已合入 `origin/main`，但根目录 `VERSION` 仍为 `0.3.5`，仓库也没有历史 `v0.3.5` 正式 tag。当前 UTM 环境运行的是 `/home/kylin/info-serve` 下的 0.3.5 部署，实际代码基线为 `f632f96`，并已使用拆分后的五个 portal 服务。因此本次 0.3.6 正式更新只交付从该已验证基线到 `origin/main@3f8b19d4` 的实际运行时变化：门户前端静态资源。

本次不修改数据库，不重建后端镜像，不重新发布 Nacos。`deploy/updates/MANIFEST.md` 中 0.3.6 的服务拆分配置已存在于当前 UTM 基线；正式报告必须记录这一环境前提，避免把该增量包用于更早的、尚未完成服务拆分的安装。

## 2. 目标与非目标

### 2.1 目标

1. 将根目录 `VERSION` 定版为 `0.3.6`。
2. 从干净的 release commit 构建 `plus-ui/dist`。
3. 生成带版本、提交号、文件清单和 SHA-256 sidecar 的正式更新包。
4. 升级前自动备份当前 `plus-ui/dist` 与 `VERSION`。
5. 通过临时目录和同文件系统重命名完成静态资源切换，避免半更新状态。
6. 升级后强制重建 `nginx-web`，并执行 Web 与网关健康检查。
7. 任一升级步骤失败时自动调用回退脚本；回退脚本也可独立人工执行。
8. 在 UTM/Kylin ARM64 上验证正常升级、故障自动回退、人工回退和最终升级。

### 2.2 非目标

- 不发布数据库 SQL。
- 不修改 Nacos 配置。
- 不替换后端 JAR 或 Docker 镜像。
- 不包含 12+9 应用离线包。
- 不在 `releases/` 中提交二进制产物。

## 3. 发布策略

采用两阶段不可变发布：

1. 从最新 `origin/main` 创建 `release/0.3.6`。
2. 提交版本号、脚本、文档与清单修订。
3. 在 release commit 上创建并推送 `v0.3.6-rc.1`。
4. 从该 commit 构建一次更新包并在 UTM 完成全部验证。
5. 验证通过后在同一 commit 上创建并推送 `v0.3.6`。
6. 已测试归档原样作为正式产物，不因正式 tag 再次构建。

这样既保留“从 tag 构建”的可追溯性，也避免在测试失败时留下指向坏提交的正式 tag。

## 4. 产物结构

正式归档命名：

```text
info-serve-update-0.3.6-<short-sha>-arm64.tar.gz
info-serve-update-0.3.6-<short-sha>-arm64.tar.gz.sha256
```

归档内根目录：

```text
info-serve-update-0.3.6/
├── RELEASE-METADATA.env
├── MANIFEST.sha256
├── README.md
├── payload/
│   └── plus-ui/
│       └── dist/
└── scripts/
    ├── apply-update.sh
    └── rollback-update.sh
```

`RELEASE-METADATA.env` 只允许简单的 `KEY=value`，包含：

- `PACKAGE_VERSION=0.3.6`
- `MIN_SOURCE_VERSION=0.3.5`
- `SOURCE_COMMIT=<完整提交号>`
- `SOURCE_SHORT_COMMIT=<短提交号>`
- `BUILD_ARCH=arm64`
- `BUILD_TIME_UTC=<UTC 时间>`

`MANIFEST.sha256` 覆盖归档根目录内除自身外的全部文件。外部 `.sha256` sidecar 覆盖压缩归档本身。

## 5. 脚本职责

### 5.1 `package-update.sh`

位置：`deploy/scripts/package-update.sh`。

职责：

1. 要求 Git 工作区干净，`VERSION=0.3.6`，当前提交可解析。
2. 要求 `plus-ui/dist/index.html` 已存在。
3. 将前端产物、升级脚本、回退脚本和 README 复制到临时装配目录。
4. 生成元数据和内部 SHA-256 清单。
5. 生成 tar.gz 与外部 SHA-256 sidecar。
6. 默认输出到仓库外或被忽略的 `releases/` 目录，不把归档加入 Git。

### 5.2 `apply-update.sh`

调用形式：

```bash
sudo ./scripts/apply-update.sh --target /home/kylin/info-serve
```

执行顺序：

1. 校验参数、目标目录、Docker/Compose、包内清单和源版本。
2. 获取排他锁，阻止并发升级或回退。
3. 创建 `.update-backups/0.3.6-<UTC 时间>-<short-sha>/`。
4. 备份 `plus-ui/dist`、`VERSION`，并写入备份元数据和 SHA-256。
5. 将新 dist 复制到目标目录同文件系统的 staging 目录。
6. 将旧 dist 重命名为事务临时目录，再把 staging 原子切换为正式 dist。
7. 写入目标 `VERSION=0.3.6` 和 `.release-info`。
8. 执行 `docker compose --env-file .env up -d --force-recreate nginx-web`。
9. 检查：
   - `http://127.0.0.1:7010/` 返回 2xx/3xx；
   - `http://127.0.0.1:7010/prod-api/auth/code` 返回 HTTP 200；
   - `nginx-web` 容器为 running。
10. 任一步失败则调用同包 `rollback-update.sh --backup <本次备份>`，并以非零码退出。

升级成功后保留备份，不自动删除。

### 5.3 `rollback-update.sh`

调用形式：

```bash
sudo ./scripts/rollback-update.sh \
  --target /home/kylin/info-serve \
  --backup latest
```

职责：

1. 校验目标目录和指定备份；`latest` 解析到时间最新的完整备份。
2. 校验备份清单；不接受缺失或哈希不匹配的备份。
3. 获取与升级脚本相同的排他锁。
4. 恢复 `plus-ui/dist`、`VERSION` 和原 `.release-info`（若存在）。
5. 强制重建 `nginx-web`。
6. 执行与升级相同的健康检查。
7. 记录回退时间、来源备份与结果，不删除失败版本或备份证据。

## 6. 错误处理与安全约束

- 脚本使用 `set -Eeuo pipefail`，错误不得静默吞掉。
- 所有路径先做绝对化和目录边界检查，拒绝 `/`、空路径和非 info-serve 目标。
- 不读取、打印或打包 `.env` 内容。
- 不执行来源不明的包；内外两层 SHA-256 均必须验证。
- 不使用 `rm -rf` 清理用户给定路径；仅清理脚本自行创建且通过前缀校验的 staging 目录。
- 升级与回退共用排他锁。
- 自动回退失败时保留现场并输出手工恢复命令，禁止继续覆盖。

## 7. UTM 验证方案

目标环境：`kylin@192.168.64.5`，`aarch64`，Kylin V10 SP1，Docker 26.1.3，Compose 5.2.0。

### 7.1 升级前基线

- 记录 `VERSION`、`dist/index.html` SHA-256、Web 容器 ID/启动时间。
- `docker compose config --quiet`。
- 冒烟检查首页、验证码接口和资料共享路由。

### 7.2 故障自动回退

使用脚本测试参数将健康检查 URL 指向确定失败的本地端口，触发升级后健康检查失败。验收：

- 升级命令非零退出；
- 自动回退被调用；
- `VERSION` 恢复为 `0.3.5`；
- `dist/index.html` SHA-256 与升级前一致；
- Web 和验证码接口恢复；
- 备份和失败日志保留。

该测试参数只用于验证，不写入正式默认值。

### 7.3 人工回退

先正常升级到 0.3.6，再执行 `rollback-update.sh --backup latest`。验收旧版本、旧哈希与服务健康全部恢复。

### 7.4 最终升级

再次执行正常升级并保留 0.3.6：

- `VERSION=0.3.6`；
- 目标 dist 与包内 dist 清单一致；
- 首页、验证码接口、资料共享页与文件预览路由可访问；
- 浏览器控制台无新增错误；
- 所有原有核心容器仍运行；
- 最近日志无新增 `ERROR`、`Exception`、`FATAL`、`panic`。

## 8. 本地与发布门禁

提交与定版前必须通过：

```bash
git diff --check
cd plus-ui && npm run design:audit
cd plus-ui && npm run build:prod
bash -n deploy/scripts/package-update.sh
bash -n deploy/scripts/apply-update.sh
bash -n deploy/scripts/rollback-update.sh
docker compose --env-file deploy/.env -f deploy/docker-compose.yml config --quiet
```

脚本还需覆盖以下本地测试：参数缺失、错误版本、清单篡改、备份缺失、锁冲突、健康检查失败触发自动回退。

## 9. 交付与版本控制

1. 提交信息使用 conventional 中文格式，不附署名尾注。
2. 推送 `release/0.3.6`、`v0.3.6-rc.1` 与验证通过后的 `v0.3.6`。
3. 合并前检查 `git log origin/main..HEAD` 与 `git diff origin/main...HEAD`。
4. `releases/` 产物永不入库。
5. 最终报告包含包路径、大小、SHA-256、UTM 命令与结果、回退证据、访问地址和残余风险。

## 10. 验收标准

- 正式 `v0.3.6` 与经过 UTM 验证的 release commit 一致。
- 更新包可在当前 0.3.5 UTM 基线上升级到 0.3.6。
- 健康检查失败能自动恢复到升级前文件和版本。
- 独立回退脚本能再次恢复旧版本且服务健康。
- 最终 UTM 状态停留在 0.3.6。
- 更新归档与 sidecar 哈希一致，Git 工作区不包含发布二进制。
