# 0.3.6 正式更新包、UTM 升级验证与失败回退设计

**状态：已确认**
**确认日期：2026-07-16**
**发布分支：`release/0.3.6`**
**基线：`origin/main@3f8b19d4`**
**目标版本：`0.3.6`**

## 1. 背景与发布边界

PR #35 已合入 `origin/main`，但根目录 `VERSION` 仍为 `0.3.5`，仓库也没有历史 `v0.3.5` 正式 tag。当前 UTM 环境运行的是 `/home/kylin/info-serve` 下的 0.3.5 部署，实际代码基线为 `f632f96`，并已使用拆分后的五个 portal 服务。仓库审计确认 `f632f96..3f8b19d4` 的非文档运行时变化全部位于 `plus-ui`，因此本次 0.3.6 正式更新只交付门户前端静态资源。

该包不是对任意标称 `0.3.5` 环境开放的通用更新，而是 **当前单台 UTM 现场快照指纹专用正式增量**。`f632f96` 是该部署的来源记录，不作为安装器无法独立证明的兼容性断言。安装器必须在任何写操作前逐项核对只读采集的 baseline receipt：旧 dist 完整指纹、Compose 文件哈希、gateway 与五个 portal 容器 image ID、关键 Nacos data-id 内容哈希、数据库 schema 标记和业务 API 探测结果；若目标仍有 Git checkout，还必须验证 `git -C "$TARGET" rev-parse HEAD`。任一证据不匹配即拒绝安装。本次不修改数据库，不重建后端镜像，不重新发布 Nacos。其他机器和更早安装不能使用本包。

## 2. 目标与非目标

### 2.1 目标

1. 将根目录 `VERSION` 定版为 `0.3.6`。
2. 从干净的 release commit 构建 `plus-ui/dist`。
3. 生成带版本、提交号、文件清单和 SHA-256 sidecar 的正式更新包。
4. 升级前自动备份当前 `plus-ui/dist` 与 `VERSION`。
5. 通过临时目录和同文件系统重命名完成静态资源切换，避免半更新状态。
6. 升级后强制重建 `nginx-web`，并执行 Web 与网关健康检查。
7. 任一升级步骤失败时自动调用回退脚本；回退脚本也可独立人工执行。
8. 在 UTM/Kylin ARM64 上验证正常升级、故障自动回退、人工回退和最终升级。

### 2.2 非目标

- 不发布数据库 SQL。
- 不修改 Nacos 配置。
- 不替换后端 JAR 或 Docker 镜像。
- 不包含 12+9 应用离线包。
- 不在 `releases/` 中提交二进制产物。

## 3. 发布策略

采用两阶段不可变发布，目标源码基线固定为 `origin/main@3f8b19d4`，不得在发布分支静默合入更新后的主线：

1. 从最新 `origin/main` 创建 `release/0.3.6`。
2. 提交版本号、脚本、文档与清单修订。
3. 推送 release 分支并创建 release PR；在 release commit 上创建并推送 `v0.3.6-rc.1`。
4. 在干净的 RC tag checkout 中重建前端和候选更新包，并在 UTM 完成全部升级、自动回退、人工回退和最终升级验证。
5. 将验证证据补入 PR，经用户 review 后使用 merge commit 合入 `origin/main`，禁止 squash，确保已测试 release commit 是主线祖先。
6. 验证 `git merge-base --is-ancestor <release-commit> origin/main` 后，在同一 release commit 创建并推送 `v0.3.6`。
7. 必须在干净的 `v0.3.6` checkout 中重新执行构建与装配。正式包的 payload dist manifest、脚本 manifest 和 source commit 必须与 RC 包逐项一致；允许 `SOURCE_TAG`、`BUILD_TIME_UTC` 和归档 SHA 不同。
8. 将 UTM 从 RC 最终态人工回退到 0.3.5，使用正式 `v0.3.6` 归档再次升级并执行最小完整冒烟，最终停留在 0.3.6。
9. 将正式归档 SHA-256、tag commit 和 RC/正式内容一致性证据发布到对应 GitHub Release。

两个 tag 指向完全相同的 release commit，但正式产物严格从 `v0.3.6` tag 重新装配并复测，符合 AGENTS/RUNBOOK 的定版顺序。验证证据只更新 PR 描述或外部附件，不在 RC tag 后追加 tracked commit；任何代码、脚本或文档提交都必须创建新的 RC、重建和重测。若主线在此期间漂移，不改变本次 release commit；任何希望纳入的新变化必须重新审计增量并重新走 RC。

## 4. 产物结构

正式归档命名：

```text
info-serve-update-0.3.6-<short-sha>-arm64.tar.gz
info-serve-update-0.3.6-<short-sha>-arm64.tar.gz.sha256
```

归档内根目录：

```text
info-serve-update-0.3.6/
├── RELEASE-METADATA.env
├── BASELINE-RECEIPT.env
├── MANIFEST.sha256
├── README.md
├── payload/
│   └── plus-ui/
│       └── dist/
└── scripts/
    ├── apply-update.sh
    ├── collect-baseline-receipt.sh
    ├── rollback-update.sh
    ├── update-common.sh
    └── verify-package.sh
```

`RELEASE-METADATA.env` 只允许简单的 `KEY=value`，包含：

- `PACKAGE_VERSION=0.3.6`
- `REQUIRED_BASE_VERSION=0.3.5`
- `REQUIRED_BASE_COMMIT=f632f960b5d3fd1737b658e45fe43b15dd319e19`
- `REQUIRED_BASELINE_ID=utm-kylin-20260716-pre-0.3.6`
- `REQUIRED_BASE_DIST_MANIFEST_SHA256=<旧 dist 排序清单哈希>`
- `SOURCE_TAG=<v0.3.6-rc.N 或 v0.3.6>`
- `SOURCE_COMMIT=<完整提交号>`
- `SOURCE_SHORT_COMMIT=<短提交号>`
- `RELEASE_BRANCH_BASE_COMMIT=3f8b19d4bb87dcf02b36058a73be7991204350a8`
- `NODE_VERSION=<构建 Node 版本>`
- `NPM_VERSION=<构建 npm 版本>`
- `PACKAGE_LOCK_SHA256=<plus-ui/package-lock.json 哈希>`
- `DIST_MANIFEST_SHA256=<新 dist 排序清单哈希>`
- `BUILD_ARCH=arm64`
- `BUILD_TIME_UTC=<UTC 时间>`

`BASELINE-RECEIPT.env` 是从 UTM 只读采集并冻结的现场凭据。`collect-baseline-receipt.sh` 是唯一采集入口，调用 `update-common.sh` 中无副作用的 `collect_baseline_receipt` 函数；打包前采集和安装前复核必须复用同一实现。输出固定为按 key 排序的 UTF-8 `KEY=hex-or-enum`，schema 如下：

- `RECEIPT_SCHEMA_VERSION=1`
- `TARGET_GIT_PRESENT=0|1` 与 `TARGET_GIT_HEAD=<sha|ABSENT>`；现场基线无 `.git` 时固定为 `0/ABSENT`，安装复核时 presence 变化也视为不匹配。
- `DIST_MANIFEST_SHA256=<sha>`。
- `COMPOSE_PORTAL_KERNEL_SHA256`、`COMPOSE_PORTAL_APPCENTER_SHA256`、`COMPOSE_PORTAL_FORUM_SHA256`、`COMPOSE_PORTAL_REQUIREDKNOWLEDGE_SHA256`、`COMPOSE_PORTAL_RESOURCES_SHA256`。
- `IMAGE_GATEWAY_ID` 以及五个 `IMAGE_PORTAL_*_ID`，取 `docker inspect .Image` 的不可变 image ID，不取可变 tag。
- Nacos `dev` 与 `prod` namespace、`DEFAULT_GROUP` 下 `portal-kernel.yml`、`portal-appcenter.yml`、`portal-forum.yml`、`portal-requiredknowledge.yml`、`portal-resources.yml`、`ruoyi-gateway.yml` 共 12 个内容 SHA-256。采集通过 OpenAPI，只接受调用方环境提供的凭据，缺项或读取失败为硬失败，禁止把凭据或配置正文写入 receipt/日志。
- `DB_SCHEMA_MARKERS_SHA256=<sha>`，由只读查询规范化后的排序行计算：`portal_module`、`info_resource_note`、`info_resource_view_record`、`info_resource_category_link` 表存在性，以及 `info_resource_category.parent_id` 列的类型/可空性。只记录 schema，不记录业务行数或内容。
- `API_CAPABILITIES_SHA256=<sha>`，只哈希稳定的规范化能力元组：验证码端点 HTTP/JSON code/captchaEnabled 类型；栏目树、分类和资料分页端点的 HTTP/鉴权 code；`/portal/resources` 与 `/portal/resources/preview/1` 的 HTTP/content-type。忽略 UUID、验证码图片、消息文本、时间、计数和业务数据。

所有 key、缺失值和失败语义在 collector 中固定：必需证据缺失直接非零退出，不输出部分 receipt。安装器逐项重新采集并比较整个 receipt，不能仅验证容器名称或 running 状态。

所有 dist 清单使用 `update-common.sh` 中唯一的 canonical 算法：在目标根目录内仅选择普通文件，输出 `./` 相对路径，使用 NUL 分隔与 `LC_ALL=C sort -z` 固定顺序，并使用 `sha256sum -z` 等价的 NUL 结尾记录；采集、装配、升级和回退必须调用同一函数。换行和其他特殊文件名不会破坏解析。`MANIFEST.sha256` 覆盖归档根目录内除自身外的全部文件。外部 `.sha256` sidecar 覆盖压缩归档本身。README 的第一步必须要求用户在解压、执行任何包内脚本前运行 `sha256sum -c <archive>.sha256`；解压后由独立、只读的 `verify-package.sh` 校验内部清单，不能把升级脚本自验当成执行前信任根。

## 5. 脚本职责

### 5.1 `package-update.sh`

位置：`deploy/scripts/package-update.sh`。

职责：

1. 要求位于干净的 RC 或正式 tag checkout，`VERSION=0.3.6`，HEAD 与 tag 可解析；执行 `git merge-base --is-ancestor 3f8b19d4 HEAD`，并要求 `git merge-base 3f8b19d4 HEAD` 精确等于 `3f8b19d4`。
2. 删除旧 `plus-ui/dist`，执行锁文件一致安装和构建（`npm ci`、测试、`lint:eslint`、design audit、production build），禁止复用陈旧 dist。
3. 装配前后再次确认 Git 工作区干净，并分别核对 `f632f96..SOURCE_COMMIT` 的运行时差异与 `3f8b19d4..HEAD` 的 release 差异只落在各自批准 allowlist。
4. 调用 `collect-baseline-receipt.sh` 从目标 UTM 只读采集完整 receipt；只有 collector 原子成功输出全部 schema 字段才允许写入包内元数据。
5. 将前端产物、升级脚本、回退脚本、纯函数共享库、独立校验器和 README 复制到临时装配目录。
6. 记录 tag、完整 commit、固定 base commit、Node/npm 版本、lockfile SHA 与新旧 dist manifest SHA。
7. 生成内部 SHA-256 清单、tar.gz 与外部 SHA-256 sidecar。
8. 默认输出到被忽略的 `releases/` 目录，不把归档加入 Git。

### 5.2 `apply-update.sh`

调用形式：

```bash
sudo ./scripts/apply-update.sh --target /home/kylin/info-serve
```

执行顺序：

1. 校验参数、目标目录、Docker/Compose 和源版本；要求调用者已成功运行独立包校验器。
2. 在加锁和任何写操作前调用同一 `collect_baseline_receipt`，要求 `VERSION=0.3.5` 且新采集 receipt 与包内 receipt 字节级一致。目标 Git presence/HEAD、旧 dist、容器 image ID、Compose/Nacos、数据库 schema 和稳定 API 能力任一不匹配均拒绝。若目标已有匹配 `REQUIRED_BASELINE_ID` 的 `.release-info` 只作为辅助证据，不能替代 receipt。
3. 获取排他锁，阻止并发升级或回退；锁冲突必须快速失败。
4. 创建 `.update-backups/0.3.6-<UTC 时间>-<short-sha>/`，用 `cp -a` 备份 `plus-ui/dist`、`VERSION` 和原 `.release-info`。元数据明确记录 `.release-info` 原本是否存在。
5. 生成覆盖备份全部普通文件、但明确排除清单自身和完成哨兵的 canonical SHA-256 清单；校验成功后最后原子创建 `BACKUP_COMPLETE`。哨兵有意不纳入清单，不完整备份不得参与 `latest` 选择。
6. 将新 dist 复制到目标目录同文件系统的 staging 目录；VERSION 和 `.release-info` 也通过临时文件加 rename 写入。
7. 将旧 dist 重命名为事务临时目录，再把 staging 原子切换为正式 dist。
8. 执行 `(cd "$TARGET/deploy" && docker compose --env-file .env -f docker-compose.yml up -d --no-deps --force-recreate nginx-web)`，记录旧/新容器 ID 与 StartedAt。
9. 检查：
   - `http://127.0.0.1:7010/` 返回 2xx/3xx；
   - `http://127.0.0.1:7010/prod-api/auth/code` 返回 HTTP 200；
   - `nginx-web` 容器为 running。
10. 备份完成后的任一步失败进入一次性错误处理：先 `trap - ERR`，设置 `ROLLBACK_IN_PROGRESS=1`，由当前持锁进程调用不再次加锁的内部 `restore_from_backup`。禁止子进程重新获取同一锁，禁止递归回退。自动回退成功仍以原升级错误码非零退出；回退失败使用独立退出码并保留现场。

升级成功后保留备份，不自动删除。

### 5.3 `rollback-update.sh`

调用形式：

```bash
sudo ./scripts/rollback-update.sh \
  --target /home/kylin/info-serve \
  --backup latest
```

职责：

1. 校验目标目录和指定备份；`latest` 只解析到时间最新、具有 `BACKUP_COMPLETE` 且清单校验通过的备份。
2. 获取与升级脚本相同的排他锁；锁冲突快速失败。
3. apply 与 rollback 都 source `update-common.sh`。该文件只定义 canonical manifest、正式健康检查和无锁 `restore_from_backup` 等无副作用函数，被 source 时不得执行入口逻辑。rollback 调用该函数，使用同文件系统 staging+rename 原子恢复完整 dist；VERSION 与 `.release-info` 使用临时文件+rename；若备份元数据表明原 `.release-info` 不存在，则删除升级新建的该文件。
4. 执行 `(cd "$TARGET/deploy" && docker compose --env-file .env -f docker-compose.yml up -d --no-deps --force-recreate nginx-web)`，记录旧/新容器 ID 与 StartedAt。
5. 执行写死的正式健康检查，不继承故障注入状态。
6. 记录回退时间、来源备份与结果，不删除失败版本或备份证据。

## 6. 错误处理与安全约束

- 脚本使用 `set -Eeuo pipefail`，错误不得静默吞掉。
- 所有路径先做绝对化和目录边界检查，拒绝 `/`、空路径和非 info-serve 目标。
- 不读取、打印或打包 `.env` 内容。
- 不执行来源不明的包；内外两层 SHA-256 均必须验证。
- 不使用 `rm -rf` 清理用户给定路径；仅清理脚本自行创建且通过前缀校验的 staging 目录。
- 升级与独立回退共用排他锁，但自动回退由持锁的升级进程直接调用无锁恢复函数。
- 自动回退失败时保留现场并输出手工恢复命令，禁止继续覆盖。
- 升级与回退写入 `$TARGET/.update-logs/<operation-id>.log` 和结构化结果元数据；日志禁止包含 `.env`，最终报告记录日志 SHA-256。

## 7. UTM 验证方案

目标环境：`kylin@192.168.64.5`，`aarch64`，Kylin V10 SP1，Docker 26.1.3，Compose 5.2.0。

### 7.1 升级前基线

- 记录 `VERSION`、完整 dist 排序清单及其 SHA-256、Web 容器 ID/启动时间，并生成精确基线指纹。
- `docker compose config --quiet`。
- 冒烟检查首页、验证码接口和资料共享路由。

### 7.2 故障自动回退

仅在显式设置 `INFO_SERVE_ALLOW_FAILURE_INJECTION=1` 时允许 `--test-fail-after-switch`。该开关只在备份完成、dist/VERSION 切换和 nginx 重建后注入一次失败；自动与人工回退始终使用写死的 7010 首页和验证码检查，不读取故障注入参数。验收：

- 升级命令非零退出；
- 自动回退被调用；
- `VERSION` 恢复为 `0.3.5`；
- 完整 dist 排序清单与升级前一致；
- Web 和验证码接口恢复；
- 备份和失败日志保留。

故障注入必须写入审计日志，正式默认不可触发。

### 7.3 人工回退

先正常升级到 0.3.6，再执行 `rollback-update.sh --backup latest`。验收旧版本、旧哈希与服务健康全部恢复。

### 7.4 最终升级

再次执行正常升级并保留 0.3.6：

- `VERSION=0.3.6`；
- 目标 dist 与包内 dist 清单一致；
- 首页、验证码接口、资料共享页与文件预览路由可访问；
- 浏览器控制台无新增错误；
- 所有原有核心容器仍运行；
- 从升级开始 UTC 时间执行 `docker logs --since <UTC>`，确认 nginx、gateway 和五个 portal 服务无新增未解释的 `ERROR`、`Exception`、`FATAL`、`panic`；既有日志不计入本次失败。

Receipt 测试必须包含一次完整正例，并分别篡改旧 dist、每类 Compose 哈希、gateway/五个 portal image ID、每类 Nacos 内容、DB schema marker、API capability 和 Git HEAD/presence，断言全部在加锁与任何写入前失败。API 测试要证明 UUID、消息和计数变化不影响规范化结果，而 HTTP/code/type/schema 变化会失败。collector 缺证据时不得留下部分输出。仅 `bash -n` 不算行为验证。

## 9. 交付与版本控制

1. 提交信息使用 conventional 中文格式，不附署名尾注。
2. 推送 `release/0.3.6`，创建 release PR，并推送 `v0.3.6-rc.1`。
3. UTM 验证后把证据补入 PR；用户 review 后使用 merge commit 合入主线。
4. 合并前检查 `git log origin/main..HEAD` 与 `git diff origin/main...HEAD`；合并后验证 release commit 是 `origin/main` 祖先，再推送 `v0.3.6`。
5. 从 `v0.3.6` 干净 checkout 重建正式包，比较其 payload/script manifests 与 RC 一致；将 UTM 回退后使用正式包重升并验证。
6. 在 GitHub Release 发布正式归档 SHA-256、tag commit 与 RC/正式内容一致性结果；README 要求同时核对这些证据。
7. `releases/` 产物永不入库。
8. 最终报告包含包路径、大小、SHA-256、UTM 命令与结果、回退证据、访问地址、审计日志 SHA 和残余风险。

## 10. 验收标准

- 正式 `v0.3.6` 与经过 UTM 验证的 release commit 一致。
- 更新包只在通过完整 baseline receipt 校验的单台 UTM 现场快照上升级到 0.3.6；其他 0.3.5 环境明确拒绝。
- 健康检查失败能自动恢复到升级前文件和版本。
- 独立回退脚本能再次恢复旧版本且服务健康。
- 最终 UTM 状态停留在 0.3.6。
- 更新归档与 sidecar 哈希一致，Git 工作区不包含发布二进制。
