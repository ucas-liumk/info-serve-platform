# 《从项目实践到软件工厂》内部汇报演讲稿设计

日期：2026-07-14

## 1. 目标与边界

为管理与技术混合听众设计一场 45 分钟内部培训，题目为：

> 从项目实践到软件工厂：AI 时代信息化自主建设能力探索

汇报不是个人成果表彰，也不是 AI 工具演示。项目承担训练场、验证样本和参考实现三种角色；最终目标是说明单位如何形成面向业务的软件全生命周期自主建设能力，并与软件工厂结合实现敏捷、可靠、可持续的交付。

当前阶段先交付完整技术版演讲稿，PPT 后续制作。稿件允许现场按时间跳过技术展开，但底稿和未来 PPT 应尽量完整。

## 2. 核心命题

全场核心命题：

> AI 可以提高局部开发速度；只有把业务建模、架构边界、工程规约、自动验证和标准交付连成闭环，速度才能转化为组织可复用的信息化建设能力。

辅助口号：

> 人定目标，架构定边界，AI 提速度，测试给证据，平台促复用。

## 3. 三部分叙事

### 第一部分：一个真实项目如何完成全生命周期建设（约 26 分钟）

1. 开门见山介绍近期建设的信息服务门户及主要功能。
2. 从项目需求引出架构选择：单体、模块化单体、微服务、云原生。
3. 对比 RuoYi/RuoYi-Vue、RuoYi-Cloud、JeecgBoot、Yudao、RuoYi-Cloud-Plus 的定位和适配场景，说明选择是“适配目标”而非项目排名。
4. 坦率说明当前业务规模使用模块化单体也能满足；采用 RuoYi-Cloud-Plus 和云原生路线主要为了实践服务治理、架构边界和部署运维能力。
5. 按需求/Spec、架构/BC、编码、集成测试、部署和迭代的顺序讲完整生命周期。
6. 用真实踩坑解释规约为何形成：GitHub 真相源、任务分支、VERSION、PORTS、SQL 三件套、Nacos 发布、单文件限制、构建/测试/冒烟门禁。
7. 展示终态业务架构和运行架构；区分 Docker Compose 的一致打包/联调价值与 K3s 的声明式编排/自愈价值。

### 第二部分：把一次项目实践抽象为软件工厂（约 12 分钟）

1. 软件工厂不是单个低代码产品或代码生成器，而是组织的软件交付生产系统。
2. 用“需求资产—参考架构—公共组件—AI 协作—质量门禁—标准交付—反馈度量”构成完整生产链。
3. 将项目已有产物映射到软件工厂：Spec/Plan、AGENTS、BC 地图、组件底座、CI/测试、Docker/K3s、VERSION/MANIFEST/RUNBOOK。
4. 区分两条生产路径：核心系统走专业代码工程；轻量台账/调查/闭环应用可走 Budibase 等低代码路径。二者共用身份、数据、治理、测试和发布标准。
5. DataEase、Apache Hop、Budibase 是可插拔工具，不是软件工厂本身；平台必须以业务价值和可替换的契约为中心。

### 第三部分：单位未来怎么干（约 7 分钟）

1. 选择边界清晰、价值明确的内部业务试点。
2. 建立业务、产品、架构、开发、测试和运维的最小联合小组。
3. 建立第一条“黄金路径”：需求模板、项目模板、公共能力、AI 上下文、流水线、部署和度量。
4. 以小批量、真实用户反馈迭代，不以代码量或上线数量衡量成效。
5. 从项目交付逐步转向能力复用，形成软件工厂和内部产品化运营。

## 4. 听众分层

每个关键内容同时提供两层信息：

- 管理层：业务价值、风险控制、资源投入、组织能力和实施路线。
- 技术人员：框架选择、BC 边界、通信契约、测试门禁、部署拓扑和工程证据。

专业术语第一次出现时必须用业务语言解释；后续再给技术细节。技术展开标为“可跳过”，以便现场根据时间裁剪。

## 5. 事实与引用纪律

- 项目事实以 `origin/main`、README、架构法源、Spec/Plan、RUNBOOK、PORTS、MANIFEST 和 Git 历史为准。
- 当前主线已经完成门户五个 BC 的独立服务拆分，并保留 Docker Compose 开发/验证路径。
- K3s 的目标架构已完成设计，但主线暂未发现 `deploy/k3s` 落地清单；在用户确认前只表述为“部署目标与实施方案”，不声称已经生产验证。
- 产品比较只引用各项目官方资料，不做无依据的性能、安全或社区优劣排名。
- 外部知识优先引用 ISO/IEC/IEEE 12207、NIST SSDF、Agile Manifesto、CNCF、Kubernetes/K3s、Docker、DORA、Microsoft DDD 指南和 Martin Fowler。

## 6. 表达设计

- 开场采用赋能承诺，不以个人经历或致谢开场。
- 第一部分结束用“标准化产物连起来就是软件工厂”完成转场。
- 第二部分结束用“平台能力只有进入真实项目才有价值”转入未来路线。
- 每 5—7 分钟回扣一次核心命题。
- 结尾保留贡献和行动方向，不以“谢谢”或“Questions?”作为最后一页。

## 7. 交付物

演讲稿文件应包含：

1. 总时间分配和删减策略。
2. 可直接朗读的逐段讲稿。
3. 现场转场语、提问点和停顿提示。
4. 技术展开与“可跳过”标记。
5. PPT 页面建议，但不在本阶段生成 PPTX。
6. 权威资料引用和项目证据索引。

## 8. 验收标准

- 三部分逻辑连续，第二部分由第一部分自然抽象而来。
- 45 分钟可讲完主线；技术展开允许形成 55—60 分钟完整版。
- 不把微服务、云原生、低代码和 AI 绝对化。
- 不把设计状态表述为已部署状态。
- 管理层能听懂价值和实施路线，技术人员能看到真实机制与证据。
- 稿件能直接转化为后续 PPT 的逐页内容和演讲者备注。
