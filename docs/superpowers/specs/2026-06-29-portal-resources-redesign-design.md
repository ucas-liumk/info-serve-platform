# 资源中心高保真重构设计

## 背景

当前 `portal/resources` 页面更接近内部资料列表：顶部标题与搜索、分类标签、三列资料卡、详情抽屉、上传弹窗。用户提供的参考图 `pics/resurces.png` 更像资源发现门户：顶部搜索、左侧多维筛选、右侧筛选栏、网格/列表切换、资源卡片和分页。

本次重构采用“系统融合型”方向：参考图的信息架构和资源发现效率，但视觉语言统一到信息中心数智服务平台现有门户风格，不直接复制外部资源站气质。

## 目标

- 把资源共享页升级为“资源中心”，支持快速发现、筛选、预览、下载和上传资料。
- 增加“我的资料”视图，让用户能管理自己上传的文件，形成上传后的闭环。
- 统一 `portal/home`、`portal/tools` 的蓝白门户风格，保持专业、克制、可扫读。
- 保留现有接口能力，明确必须补充的最小后端能力，避免一次性大改。
- 普通用户不依赖后台管理权限即可上传和维护自己的资料。

## 非目标

- 不重做门户首页、工具页、论坛页。
- 不引入新的独立设计系统。
- 不做复杂评分体系；参考图中的星级可转化为更适合内部平台的热度/下载/浏览指标。
- 不把普通用户引到后台管理页完成个人资料管理。

## 信息架构

页面包含两个一级视图：

1. `资源中心`
   面向所有可见资源的发现和使用。

2. `我的资料`
   面向当前登录用户上传资料的管理。

两个视图共享页面顶部品牌、搜索、上传入口和基础视觉框架，但内容区的重点不同：资源中心偏发现，我的资料偏管理。

## 视觉设计

整体风格使用现有门户的蓝白基调：

- 背景：浅蓝白渐变或轻质感背景，避免大面积深色和复杂装饰。
- 主色：沿用门户蓝，强调搜索按钮、激活状态、主要操作。
- 圆角：普通卡片和控件以 8px 为主，少量容器可用 12px。
- 阴影：轻阴影，突出层次但不制造重浮层。
- 字体：页面标题、卡片标题、筛选项都保持紧凑可扫，不使用过大的营销式标题。
- 图标：优先使用 Element Plus Icons，不手绘 SVG。文件类型使用统一颜色块或小徽标表示。

参考图中的封面卡片不直接照搬。若资源没有真实封面，卡片使用“格式色块 + 文件类型 + 标题摘要”的统一封面样式，保证内部资料也能呈现高保真质感。

## 页面布局

### 顶部区域

顶部包含：

- 平台标识或资源中心标题区。
- 主搜索框：搜索标题、简介、文件名。
- `上传资料` 按钮。
- `我的资料` 入口。
- 登录/用户状态入口沿用现有系统能力。

搜索框在桌面端居中偏宽，移动端占满一行。

### 主体区域

主体采用左侧筛选栏 + 右侧内容区：

- 左侧宽度约 230-260px。
- 右侧为主内容卡片区。
- 1366px 以上保持左右布局。
- 1024px 以下左侧筛选收敛为横向筛选区或抽屉。
- 760px 以下优先单列展示资源卡片。

## 资源中心视图

### 左侧筛选

左侧筛选包含：

- 资源分类：来自 `listResourceCategories()`，展示分类名称和数量。
- 文件类型：全部、文档、PPT、表格、PDF、图片、视频、音频、压缩包、其他。
- 上传时间：全部、最近一天、最近一周、最近一个月。

分类筛选继续真实请求后端。文件类型和上传时间需要后端补充查询条件后才能完全真实生效；前端先按完整状态设计，参数结构预留。

### 右侧筛选栏

右侧顶部包含：

- 排序：最新上传、下载最多、浏览最多。
- 格式快捷筛选：全部格式、文档、图片、视频等。
- 大小筛选：全部大小、小于 10MB、10-100MB、大于 100MB。
- 视图切换：网格视图、列表视图。

排序和大小属于增强筛选。若后端暂未支持，先完成 UI 状态与参数传递设计，再补后端落地。

### 网格卡片

资源卡片包含：

- 文件类型徽标或轻封面。
- 分类名称。
- 文件后缀。
- 标题。
- 简介或原始文件名。
- 文件大小。
- 浏览数。
- 下载数。
- 创建时间或上传者。
- 操作：查看、预览、下载。

卡片悬停时可以轻微提升阴影，操作按钮保持清晰但不喧宾夺主。

### 列表视图

列表视图适合密集比较，字段包括：

- 标题和简介。
- 分类。
- 文件类型。
- 文件大小。
- 浏览数 / 下载数。
- 上传时间。
- 操作：查看、预览、下载。

列表行高度保持紧凑，移动端退化为纵向信息块。

### 详情抽屉

详情抽屉保留现有能力并优化信息层级：

- 标题、分类、文件类型。
- 简介。
- 原始文件名、文件大小、上传时间、浏览数、下载数。
- 主要操作：预览、下载。

打开详情时继续触发现有浏览量增长逻辑。

## 我的资料视图

### 入口

顶部 `我的资料` 进入同页视图，建议使用 segmented control 或按钮状态切换：

- `资源中心`
- `我的资料`

普通用户不需要进入后台管理页即可管理自己上传的文件。

### 管理列表

我的资料默认使用列表视图，字段包括：

- 资料标题。
- 分类。
- 文件类型 / 大小。
- 上传时间。
- 状态：已发布、审核中、已下架、失败。
- 浏览数 / 下载数。
- 操作：查看、预览、下载、编辑信息、重新上传、下架或删除。

若当前系统暂未引入审核流程，状态可以先映射为：

- `0`：已发布。
- 非 `0` 或被下架：不可见 / 已下架。

后续若增加审核，可平滑扩展。

### 空状态

用户没有上传资料时，展示：

- 空状态说明：还没有上传资料。
- 主按钮：上传资料。
- 辅助说明：上传后可在这里查看发布状态、下载量和管理操作。

## 上传流程

上传弹窗保留现有能力并升级体验：

- 标题。
- 分类。
- 简介。
- 文件拖拽上传。
- 文件选择后自动填充标题。
- 发布按钮。

上传成功后：

1. 关闭弹窗。
2. 刷新分类数量。
3. 如果当前在我的资料视图，刷新我的资料列表。
4. 如果当前在资源中心，提示“已发布，可在我的资料中管理”。

## 数据与接口设计

### 现有能力

当前前端已使用：

- `GET /infoservice/portal/resources/categories`
- `GET /infoservice/portal/resources`
- `GET /infoservice/portal/resources/{resourceId}`
- `GET /infoservice/portal/resources/{resourceId}/preview`
- `GET /infoservice/portal/resources/{resourceId}/download`
- `POST /infoservice/resource/upload`
- `POST /infoservice/resource`

注意：`POST /infoservice/resource/upload` 和 `POST /infoservice/resource` 当前属于后台资源管理接口，需要 `infoservice:resource:upload`、`infoservice:resource:add` 等权限。它们不能直接作为普通用户“我的资料”闭环的唯一依赖。

当前列表支持的核心查询能力：

- `keyword`
- `categoryCode`
- `pageNum`
- `pageSize`

实体已包含：

- `fileSuffix`
- `mimeType`
- `fileSize`
- `previewType`
- `downloadCount`
- `viewCount`
- `status`
- `createTime`

`InfoResource` 继承租户基础实体，通常具备创建人相关字段，可用于我的资料筛选。

### 必要后端补充

为了实现完整“我的资料”管理，需要最小补充：

1. 我的资料查询
   - 方案 A：`GET /infoservice/portal/resources?scope=mine`
   - 方案 B：`GET /infoservice/portal/resources/mine`
   - 推荐方案 A，便于复用现有分页与筛选结构。
   - `scope=mine` 必须按当前登录用户 `createBy` 过滤，并允许返回该用户自己的非公开状态资料；公共资源中心仍只返回 `status = 0` 的可见资料。

2. 门户侧上传与创建
   - 推荐新增 `POST /infoservice/portal/resources/upload`，复用现有文件上传服务，但只要求登录态，不要求后台管理权限。
   - 推荐新增 `POST /infoservice/portal/resources`，创建资料时由后端写入当前用户为创建人，不信任前端传入的用户身份。
   - 如果复用后台接口，必须给普通用户设计明确权限，否则上传入口对多数用户不可用。

3. 当前用户资源操作
   - `PUT /infoservice/portal/resources/{resourceId}`：编辑标题、分类、简介。
   - `PUT /infoservice/portal/resources/{resourceId}/status`：下架或重新发布自己上传的资料。
   - `DELETE /infoservice/portal/resources/{resourceId}`：删除自己上传的资料。
   - 所有操作必须服务端校验 `createBy == LoginHelper.getUserId()`，管理员后台权限不能替代普通用户所有权校验。
   - 重新上传文件可作为后续增强，不阻塞第一版。

4. 资源 VO 补充
   - `ownerName` 或 `createByName`。
   - `canManage`：当前用户是否可管理该资料。
   - 可选 `ownerId`，不建议前端依赖该字段做权限判断。

5. 列表筛选增强
   - `fileType` 或 `previewType`
   - `uploadedWithin`
   - `sizeRange`
   - `sort`

第一版若要满足“完整交互”，至少需要补齐门户侧上传、创建、我的资料查询、编辑、下架/删除接口。文件类型、时间、大小、排序筛选可以按同一轮补齐；如果延期，前端必须清楚标注为本地状态或禁用真实筛选，不得让用户误以为筛选已生效。

## 前端状态设计

主要状态：

- `activeView`: `discover` 或 `mine`
- `keyword`
- `categoryCode`
- `fileType`
- `uploadedWithin`
- `sort`
- `sizeRange`
- `displayMode`: `grid` 或 `list`
- `pageNum`
- `pageSize`
- `selectedResource`
- `detailVisible`
- `uploadVisible`

资源中心和我的资料共享筛选状态，但切换视图时重置页码。我的资料可以保留关键词、分类、文件类型筛选，默认排序为最新上传。

## 权限规则

- 未登录用户：可以浏览公开资源；上传和我的资料入口引导登录或隐藏，按现有登录策略决定。
- 普通登录用户：可以上传资料，可以管理自己上传的资料。
- 管理员：继续拥有后台全量管理能力，也可以在门户查看资源。

所有“编辑、下架、删除”权限以服务端校验为准，前端只做入口显隐。

门户侧用户操作接口只校验登录态和资源所有权，不应要求后台 `infoservice:resource:*` 管理权限；后台接口继续保留给管理员做全量治理。

## 响应式设计

- 宽屏：左筛选 + 右内容，网格 4-5 列，列表完整字段。
- 中屏：网格 3 列，筛选栏保持左侧或压缩。
- 平板：筛选变为顶部折叠区，网格 2 列。
- 手机：单列卡片，筛选使用抽屉，列表视图切换可隐藏或保留为简化列表。

所有按钮最小点击高度保持 36px，移动端不低于 44px。

## 错误、加载和空状态

- 页面加载：资源区骨架屏或 Element Plus loading。
- 列表为空：根据视图展示不同文案。
- 筛选无结果：提供“清空筛选”按钮。
- 上传失败：保留错误消息并允许重新提交。
- 下载/预览失败：显示提示，不关闭详情抽屉。

## 验收标准

- 页面首屏与参考图的信息结构一致：顶部搜索、左侧筛选、右侧筛选栏、资源卡片、分页清晰可见。
- 视觉与现有门户统一，不出现明显外部模板感。
- 搜索、分类、分页、详情、预览、下载、上传继续可用。
- 网格/列表切换可用。
- 我的资料入口存在，用户能真实看到并管理自己上传的文件。
- 不能把“我的资料真实过滤”和“文件类型/时间/大小真实筛选”误写成已完成能力；若当前版本上线这些控件，就必须接入真实接口或明确禁用。
- 桌面和移动至少各验证一个视口，不能出现文字挤压或控件重叠。

## 实施顺序建议

1. 重构 `portal/resources/index.vue` 视觉结构和前端状态。
2. 保持现有真实接口可用：分类、搜索、分页、详情、上传、预览、下载。
3. 增加资源中心/我的资料双视图 UI。
4. 增加网格/列表切换和筛选状态。
5. 补门户侧普通用户上传、创建、我的资料查询、编辑、下架/删除接口。
6. 接入真实文件类型、时间、大小、排序筛选。
7. 浏览器验证桌面与窄屏效果。
