## 1. 定位声明

> **面向信创环境的嵌入式文档能力引擎**——私有化部署的文档阅读、标注、编辑与智能处理 SDK。国际厂商（Apryse/Nutrient/Foxit）的文档 SDK 能力，在麒麟/统信 + 国产芯片 + 离线机房里原生可用，并以 AI 插件生态实现反超。

**给谁**：政企行业软件商（ISV）、系统集成商、有信创合规要求的自研团队。

**解决什么**：业务系统需要文档能力时的三难——国际 SDK 用不了（合规）/不敢用（供应链）/太贵；国内文档中台 AI 弱、插件封闭、集成体验差；开源拼装（kkFileView + pdf.js + 自研）成本高、无标注、无生态。

**产品力四支柱**：

1. **信创工程化一等公民**：麒麟/统信 × arm64 离线交付是默认形态而非兼容项；OFD 与 PDF 同级支持（ofdrw）；依赖全走宽松 license 白名单（Apache/MIT/BSD，禁 AGPL），可控可审计
2. **锚定 API**（独有资产）：阅读版本固化 + 双 selector（TextQuote 主锚 + 位置提示 + 归一化 rects），标注、AI 结果、跨文档引用精确到字回锚原文，版本升级不漂移
3. **AI 插件生态 + 模型中立**：OCR/摘要/朗读/关键词以容器插件交付；LLM 走 OpenAI-compatible Provider 抽象——客户接自己的国产大模型，产品不绑模型、不碰数据
4. **集成 DX**：一个 Web 组件 + 一个 docker compose 起全栈；官方演示工作站 = 活文档 + Demo + 首客户（info-serve）集成前验证场

**北极星场景**：「AI 说的每句话都能带你回到原文」——摘要逐句回跳高亮、扫描件 OCR 后可选可标、朗读跟随高亮。该场景同时依赖转换管线、锚定 API、AI 插件三层，竞品复制需重建全部地基。

## 2. 竞品格局摘要

| 格子 | 玩家 | 与我们的关系 |
|---|---|---|
| 免费开源预览 | kkFileView（Apache-2.0） | 集成 DX 的基线对手：无标注/无插件/无 AI |
| PDF 工具箱 | Stirling-PDF | 工具箱形态参照，非引擎/SDK |
| 商业文档 SDK | Apryse(PDFTron)/Nutrient(PSPDFKit)/Foxit | **对标对象**：功能栈标杆，被信创挡在门外 |
| 编辑 SDK | Univer（Apache-2.0 核）/OnlyOffice(AGPL)/Collabora | Univer 是编辑插件底座候选；AGPL 系不可引入 |
| 国内文档中台 | 永中 DCS/WPS 开放平台/数科 OFD/福昕国产线 | 直接商业竞品：信创强但 AI 弱、生态封闭 |
| 文档 AI (IDP) | 合合 TextIn/百度/阿里读光；MinerU/RAGFlow-deepdoc | 有 AI 没前端；AI 结果回不到人看的界面 |

市场空档 = 三格交界：开源可私有化 × 阅读标注 SDK × AI 原生（结果可回锚）。

**待联网核实**（本轮 Web 工具因环境故障不可用，按 2026-01 知识写就）：Stirling-PDF 当前 license 与商业条款；Univer 商业版边界与编辑插件可行性；永中 DCS 定价模式；kkFileView 是否已有标注能力。

| 能力 | 国际 SDK | 我们 | 期次 |
|---|---|---|---|
| 多格式转换预览（Office/PDF/OFD/图片） | ✓ | ✓ 内核 | **P0** |
| 阅读版本固化 + 版本管理（历史/回滚/标注随版本迁移） | 部分 | ✓ 内核（编辑与锚定的共同地基） | **P0** |
| 文本标注（高亮/下划线/批注）+ 锚定 API | ✓ | ✓ 内核+差异化 | **P0** |
| 前端 SDK(Web 组件) + 演示工作站 | Demo 站 | ✓ | **P0** |
| OCR 双层化（扫描件可选可搜） | ✓ | PaddleOCR 容器插件 | P1 |
| AI 工具（摘要/关键词/朗读/翻译） | 弱/无 | **插件生态，反超点**；朗读默认 Web Speech，服务端 TTS 可选 | P1–P2 |
| 图形标注（框/箭头/手绘）、书签 | ✓ | 前端插件 | P2 |

| **文档编辑（默认阅读，点击进入编辑）** | ✓ | **编辑器插件**（Univer 类底座评估；保存即产生新版本→重转换→标注重锚迁移） | P2–P3 |
| 表单填写、电子签章（国密/OFD 签章） | ✓ | 插件位预留（重合规，单独立项） | 远期 |

### 3.1 编辑与版本管理的产品规则（用户确认）

- **默认阅读态**；有编辑权限的用户点击「编辑」进入编辑态，编辑的是**源文档**（非 PDF 渲染稿）
- 保存 = 生成**新文档版本**：源文件新版本入库 → 转换管线重新固化阅读版本 → `is_current` 切换
- 旧版本永久保留：既有标注/AI 结果仍绑定旧版本可查；提供「迁移到新版本」的重锚定流程，失配进孤儿列表，**永不静默错位**
- 版本管理是 P0 内核能力（编辑插件是它的消费者，P2 才上；P0/P1 的版本来源是「重新上传/重新转换」）

## 4. 架构总览（决策级，详设待产品 spec）

```
┌─ 前端 SDK（Web Component）─────────────────────────────┐
│ 阅读器内核(pdfjs) · 标注层 · 面板宿主(插件注册表) · 事件总线  │
└──────────────────────┬────────────────────────────────┘
                       │ REST/SSE + 鉴权中立(token 回调)
┌─ 引擎服务(Java)───────┴───────────────────────────────┐
│ 文档库/版本库 · 转换管线(插拔步骤) · 锚定/标注服务 · 插件宿主 │
└───┬──────────────┬──────────────┬─────────────────────┘
    │ manifest 注册 │              │
 [LibreOffice]  [PaddleOCR 插件] [LLM 工具插件→OpenAI-compatible Provider]
 [ofdrw 步骤]   [TTS 插件(可选)]  [编辑器插件(P2, Univer 类底座)]
```

- 前端插件：官方插件随版本构建集成，运行时按许可/配置启停；第三方运行时加载列远期
- 后端插件：一律独立容器 + manifest 注册（装插件 = 部容器 + 注册）
- 鉴权中立：不内置用户体系，token 校验回调给宿主系统（info-serve 用 Sa-Token 适配器）
- 内核复用 info-serve 已验证设计：锚定模型、版本固化、转换管线插拔化、面板注册表（以解耦形态迁入）

## 6. 风险与对冲

| 风险 | 对冲 |
|---|---|
| 范围膨胀（对标国际全家桶） | 期次表是硬边界；P0 只做内核四件套；签章/表单远期不启动 |
| 编辑插件复杂度（Univer 集成/格式保真） | P2 前先做底座评估 spike；版本管理让「编辑失败」永远可回滚 |
| 信创适配碎片化（OS×芯片矩阵） | P0 只认麒麟+arm64 一条基线，其余按客户驱动扩展 |

| 单人/小团队产能 | 内核迁移复用 info-serve 已验证设计与 Plan 1 代码块，不从零写 |
