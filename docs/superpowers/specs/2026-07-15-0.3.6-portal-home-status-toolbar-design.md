# 0.3.6 门户首页右上角工具栏优化设计

## 1. 背景与范围

0.3.6 是完整版本，不只包含门户首页。版本后续还会逐个优化其他界面，并纳入已确认的 Bug 修复。本设计仅定义 0.3.6 的第一个页面增量：优化门户首页右上角日期、用户和快捷操作区域。

当前首页整体布局、长城背景、平台 Logo、模块插画、服务卡片区和统计区保持不变。用户确认现有整体布局没有明显问题，本次不以重排首页为目标。

实现必须遵守：

- `AGENTS.md`
- `docs/design/design-system.md`
- `plus-ui/src/assets/styles/tokens.scss`
- 单文件不超过 800 行
- 前端样式变更必须通过设计审计、生产构建和截图对比

## 2. 现状问题

右上角信息区由 `HomeTopbar.vue` 的 `status-panel` 实现，当前包含日期、时间、星期、用户菜单、系统使用手册和通知入口。

主要问题不是信息缺失，而是容器层级过多：

- 390px 宽的蓝色玻璃卡视觉重量较大，与左侧平台标题争夺注意力。
- 日期行和用户行通过粗分区、多个竖分割线组织，显得机械。
- 系统手册按钮在大卡内部再次使用独立卡片、边框和投影，形成“卡中卡”。
- 状态面板、头像和按钮使用多套自定义阴影及硬编码色值，不利于设计令牌收敛。
- 桌面端的装饰重量高于信息价值，未体现政企门户“克制即专业”的原则。

## 3. 已选方案

用户在三个视觉方向中选择 B：**无卡片悬浮工具栏**。

该方案移除右上角整体卡片，仅为需要明确点击边界的用户入口和图标操作提供轻量承托。日期时间作为文字信息展示，不再被包裹在独立卡片中。

## 4. 设计目标

- 降低右上角视觉重量，让平台品牌与核心服务入口重新成为首页主角。
- 在长城背景上保持日期、时间和操作入口清晰可读。
- 保留全部现有功能和事件接口，不引入后端、路由或数据结构变更。
- 统一按钮尺寸、颜色、阴影、动效与焦点状态，减少硬编码视觉值。
- 在 375px、768px、1024px 和 1440px 宽度下保持可读、可点和不横向溢出。

## 5. 组件结构

### 5.1 顶栏总体结构

`HomeTopbar.vue` 继续保留左右两组：

- 左侧 `brand`：平台 Logo、名称和副标题，不改内容和资产。
- 右侧工具组：日期时间、用户入口、系统使用手册、通知。

右侧从 `status-panel` 两行大卡改为单行 `utility-toolbar`。不新增全局组件，也不改变 `HomeTopbar` 对外的 `command` 与 `open-manual` 事件。

### 5.2 日期时间组

- 时间是主信息，使用等宽数字特性避免分钟更新时跳动。
- 日期与星期作为辅助信息，放在时间下方或相邻的次级行。
- 保留当前每 60 秒更新时间的逻辑，不增加秒级刷新。
- 日期时间仅展示，不模拟按钮，也不进入键盘 Tab 顺序。

### 5.3 用户入口

- 保留头像、昵称、下拉箭头和 Element Plus 下拉菜单。
- 保留个人信息、修改密码和退出登录命令。
- 用户入口使用轻微半透明底或 neutral soft 表面，形成明确点击边界，但不使用大投影。
- 头像缺失时继续使用现有用户图标兜底。
- 按钮必须保留原生 `button` 语义和可见 `:focus-visible` 状态。

### 5.4 快捷操作

- 系统使用手册和通知入口采用同一尺寸、同一圆角和同一交互状态。
- 桌面端点击目标至少 36px，移动端至少 44px。
- 图标按钮必须提供准确的 `title` 或 `aria-label`；通知组件现有未读语义保持不变。
- 去掉手册按钮现有的自定义深投影和上浮式装饰，仅保留令牌化 hover、pressed 与 focus 反馈。

## 6. 视觉规范

- 颜色只使用现有 `--ip-*` 设计令牌和必要的透明混合，不新增页面专用硬编码色。
- 不使用整体深色卡、渐变卡或大面积模块强调色。
- 容器需要承托时只使用 `--ip-neutral-0/50/100`、`--ip-neutral-200` 与 `--ip-shadow-sm`。
- 用户和图标按钮的状态转换使用现有动效令牌，时长控制在 120–200ms。
- 不使用会引发布局位移的 hover 效果；允许颜色、边框、背景和轻微阴影变化。
- 所有交互元素使用统一焦点环，不裸移除 outline。

## 7. 响应式行为

### ≥ 1024px

- 顶栏保持左右单行布局。
- 工具栏右对齐，日期时间、用户入口和两个快捷操作在同一视觉组内。
- 不设置固定 390px 大卡宽度，使用内容驱动宽度并限制不挤压品牌区。

### 841–1023px

- 允许缩小组间距和辅助文字，但不隐藏核心操作。
- 若横向空间不足，工具组可换至品牌区下方，保持整体左对齐或右对齐的一致规则。

### ≤ 840px

- 顶栏改为纵向排列，工具组占可用宽度。
- 用户入口可伸展，快捷图标固定在末端。
- 所有交互目标高度至少 44px。

### ≤ 520px

- 日期时间组允许两行显示；昵称过长时省略并可通过下拉入口查看账号上下文。
- 不隐藏系统手册和通知入口，不产生横向滚动。

## 8. 数据与错误处理

本次不新增请求。日期来自浏览器本地时间，用户信息来自现有 `useUserStore`，通知由现有 `PortalNotificationBell` 管理。

- 用户昵称为空时继续显示“当前用户”。
- 头像加载失败不应影响下拉菜单使用；继续由现有兜底图标保证入口可识别。
- 手册、个人信息、修改密码、退出登录和通知的错误处理沿用现有调用链，本次不改变业务行为。
- 样式优化不得吞掉或重写现有事件。

## 9. 文件范围

预计只修改：

- `plus-ui/src/views/portal/home/components/HomeTopbar.vue`

仅在发现共享通知按钮无法与工具栏一致时，才允许最小范围调整：

- `plus-ui/src/layout/portal/components/PortalNotificationBell.vue`

不修改：

- 首页其他组件和图片资产
- 后端接口与数据库
- 路由和权限模型
- 其他门户页面
- `/VERSION`（在 0.3.6 所有页面与 Bug 修复完成、进入定版时统一更新）

## 10. 验收标准

### 功能回归

- 日期、时间和星期显示正确，分钟更新正常。
- 用户下拉菜单可打开，个人信息、修改密码和退出登录命令保持可用。
- 系统使用手册可打开。
- 通知入口与未读状态保持可用。

### 视觉与可访问性

- 右上角不再出现整体蓝色玻璃大卡。
- 日期时间、用户入口和快捷操作层级清晰，无“卡中卡”。
- 桌面端和移动端点击目标符合设计系统要求。
- 键盘可依次访问用户、手册和通知入口，焦点状态清晰。
- 375px、768px、1024px、1440px 无横向溢出或遮挡。
- 使用 reduced-motion 时不依赖动画表达状态。

### 必过验证

在 `plus-ui` 目录执行：

```bash
npm run design:audit
npm run build:prod
```

样式变更前后执行首页截图对比：

```bash
python3 deploy/scripts/ui-capture.py
```

如自动截图受登录态阻塞，必须记录原因，并在相同 origin 登录后补齐首页 1440px 与 375px 人工截图验证。

## 11. 版本控制与双机流程

- 开发分支：`release/0.3.6`，基于最新 GitHub `origin/main`。
- 本页面实现使用独立提交，建议提交信息：`feat: 优化门户首页右上角工具栏`。
- 0.3.6 的其他页面优化和 Bug 修复继续使用独立提交，禁止揉成一个不可审查的大提交。
- Mac 验证完成后推送 GitHub。
- Windows 当前 `E:\gallant-dev\active\info-serve` 有其他任务的未提交改动，不切换、不覆盖；另建 `E:\gallant-dev\active\info-serve-0.3.6-home`，直接从 GitHub 同步 `release/0.3.6` 进行复验。
- 版本定版时再统一更新 `/VERSION`、打 tag，并按发布规约处理。

## 12. 后续顺序

本页面验收后，回到 0.3.6 版本清单，选择下一个界面继续“视觉方案 → 设计确认 → 独立实现 → 独立验证”。Bug 修复单独进入待办并按影响范围排序，不因为 UI 工作而被混入同一提交。
