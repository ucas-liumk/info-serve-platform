# 0.3.6 门户首页右上角工具栏实施计划

依据：`docs/superpowers/specs/2026-07-15-0.3.6-portal-home-status-toolbar-design.md`（已获用户确认并通过独立规格审查）。

分支：`feat/0.3.6-portal-home-toolbar`。对比基准：`origin/main` `f632f96`。本任务完成后不在该分支继续叠加其他页面或 Bug。

## T1 改动前基线

1. 确认工作树仅包含已经提交的规格与计划。
2. 检查可用的本地门户服务与登录态。
3. 从仓库根执行 `python3 deploy/scripts/ui-capture.py`，尽可能采集当前首页桌面与移动截图。
4. 如果脚本因服务、浏览器或登录态失败，保留完整错误输出，并在实现后使用相同条件重试；不得把登录页当作首页基线。

## T2 时间与头像行为

Owned file：`plus-ui/src/views/portal/home/components/HomeTopbar.vue`。

1. 新增首次对齐下一个整分钟的 timeout，触发后更新时间并启动 60 秒 interval；卸载时同时清理 timeout 与 interval。
2. 新增头像失败状态：有 URL 且未失败时显示图片，`error` 后切换用户图标；监听头像 URL 变化并重置失败状态。
3. 保持日期格式、用户命令和 `open-manual` 事件契约不变。

## T3 无卡片工具栏视觉

Owned file：`plus-ui/src/views/portal/home/components/HomeTopbar.vue`。

1. 将 `status-panel` 两行卡片改为 `utility-toolbar`：日期时间组、用户入口、手册、通知按 DOM 顺序排列。
2. 移除整体玻璃大卡、重复分割线、卡中卡与自定义深阴影。
3. 颜色、边框、圆角、阴影、焦点和动效只使用 `--ip-*` 令牌；透明表面只用 token 派生 `color-mix()`。
4. 桌面端单行；≤840px 纵向顶栏；≤520px 使用明确两列网格，日期和用户可收缩，图标按钮固定 44px。
5. 用户、手册、通知的视觉顺序、DOM 顺序和键盘顺序一致。

## T4 通知入口最小适配

Owned file：`plus-ui/src/layout/portal/components/PortalNotificationBell.vue`。

1. 通知按钮增加 `aria-label="通知"`。
2. popover 宽度改为响应式数值：桌面最大 500px，窄屏不超过 `window.innerWidth - 32px`；通过响应式状态处理，不依赖 Element Plus 不接受的 CSS 字符串宽度。
3. 通知按钮改用与手册按钮一致的令牌化尺寸与交互状态，移动端 44px。
4. 不修改通知请求、已读、删除、清空等数据流；其静默错误作为独立 0.3.6 Bug 任务处理。

## T5 静态与构建门禁

在 `plus-ui` 执行：

```bash
npm run lint:eslint -- src/views/portal/home/components/HomeTopbar.vue src/layout/portal/components/PortalNotificationBell.vue
npm run design:audit

```

补充检查：

```bash
git diff --check
git diff origin/main...HEAD -- plus-ui/src/views/portal/home/components/HomeTopbar.vue plus-ui/src/layout/portal/components/PortalNotificationBell.vue
```

设计审计基线只允许保持或下降，禁止上调。

## T6 视觉与交互验收

1. 从仓库根重跑 `python3 deploy/scripts/ui-capture.py`。
2. 检查 1440px 首页：右侧无整体蓝色卡，日期、用户、手册和通知层级清晰。
3. 检查 375px 首页：无横向滚动；用户昵称可省略；手册和通知为 44px；通知弹层不超过视口安全区。
4. 键盘 Tab 顺序依次为用户、手册、通知，焦点环可见。
5. 使用断链头像 URL 验证图标兜底；头像 URL 更新后重新加载。
6. 跨整分钟观察时间刷新，不继续显示上一分钟。
7. 检查 `prefers-reduced-motion: reduce` 下操作反馈不依赖动画。

## T7 提交、GitHub 与 Windows 复验

1. 源码独立提交：`feat: 优化门户首页右上角工具栏`，禁止署名尾注。
2. 自查 `git log origin/main..HEAD`、`git diff origin/main...HEAD` 和验证输出。
3. 推送 `feat/0.3.6-portal-home-toolbar` 到 GitHub，创建 Draft PR，描述使用 `Refs` 指向 Issue（如有）。
4. Windows 不触碰有未提交改动的 `E:\gallant-dev\active\info-serve`；新建 `E:\gallant-dev\active\info-serve-0.3.6-home`，远端直接使用 GitHub，并检出同名分支。
5. Windows 执行 `npm run design:audit`、`npm run build:prod`，启动或更新前端后完成 1440px 与 375px 人工验收。
6. Windows 复验结果和截图回传到 PR；用户认可前不合入主线。

## 风险与回滚

- Element Plus popover 的 `width` prop 需传数值，不能直接传 `min()` CSS；使用窗口宽度计算并监听 resize。
- 背景图明暗变化可能降低无卡片文字可读性；允许对用户入口和图标按钮使用 token 派生的轻表面，不恢复整体大卡。
- 通知组件存在既有静默错误；本任务只做展示与可访问性最小适配，后续必须用独立 Bug 分支处理。
- 回滚只需撤销本任务源码提交；规格与计划可保留为决策记录。
